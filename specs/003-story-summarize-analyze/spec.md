# Feature Specification: Story Summarize & Analyze

**Feature Branch**: `003-story-summarize-analyze`  
**Created**: 2026-03-28  
**Status**: Draft  
**Input**: User description: "Add a Story Summarize & Analyze feature for persisted stories in the catalog. This feature uses the same local LLM (via the existing LM Studio integration) and consists of three capabilities, each invoked separately from the story view: Summarize, Analyze, and Rank. Accuracy over speed — use separate, focused LLM calls rather than a single combined prompt."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Summarize a Persisted Story (Priority: P1)

As a user, I can open a persisted story from the catalog and generate a short synopsis/summary that captures the story's key narrative points. The summary is persisted alongside the story metadata so I can reference it later without regenerating.

**Why this priority**: Summary is the simplest, highest-value capability — it provides immediate utility with a single LLM call and enables users to quickly assess story content without re-reading.

**Independent Test**: Can be fully tested by opening a persisted story detail view, invoking Summarize, verifying the LLM is called with the full story text, and confirming the returned summary is displayed and persisted with the story record.

**Acceptance Scenarios**:

1. **Given** a persisted story is open in detail view, **When** the user invokes Summarize, **Then** the system sends the full story text to the local LLM and displays the generated summary.
2. **Given** a Summarize request completes successfully, **When** the summary is returned, **Then** the system persists the summary alongside the story metadata in SQLite.
3. **Given** a story already has a persisted summary, **When** the user views the story detail, **Then** the existing summary is displayed without requiring a new LLM call.
4. **Given** a story already has a persisted summary, **When** the user invokes Summarize again, **Then** the system regenerates the summary by calling the LLM and replaces the previously persisted summary.
5. **Given** the local LLM is unavailable or returns an error, **When** the user invokes Summarize, **Then** the system displays an actionable error message and does not overwrite any existing summary.

---

### User Story 2 - Analyze a Persisted Story (Priority: P2)

As a user, I can analyze a persisted story to extract structured information about characters, themes, plot structure, and writing style. Each analysis dimension is generated by a separate LLM call for accuracy with smaller local models, and results are persisted alongside the story metadata.

**Why this priority**: Analysis builds on the same LLM integration as Summarize but delivers richer structured insights, making it the natural second step.

**Independent Test**: Can be fully tested by opening a persisted story, invoking Analyze, verifying four separate LLM calls are made (characters, themes, plot structure, writing style), and confirming all results are displayed and persisted.

**Acceptance Scenarios**:

1. **Given** a persisted story is open in detail view, **When** the user invokes Analyze, **Then** the system sends separate LLM requests for each analysis dimension: characters, themes, plot structure, and writing style.
2. **Given** all four analysis requests complete successfully, **When** results are returned, **Then** the system displays each dimension's results in distinct sections and persists them alongside the story metadata.
3. **Given** a story already has persisted analysis results, **When** the user views the story detail, **Then** the existing analysis is displayed without requiring new LLM calls.
4. **Given** a story already has persisted analysis, **When** the user invokes Analyze again, **Then** the system regenerates all four dimensions and replaces the previously persisted analysis.
5. **Given** one or more analysis dimension calls fail, **When** the remaining calls succeed, **Then** the system displays and persists successful results and shows actionable error messages for failed dimensions without discarding partial results.
6. **Given** the local LLM is unavailable, **When** the user invokes Analyze, **Then** the system displays an actionable error message and does not overwrite any existing analysis.

---

### User Story 3 - Configure and Manage Ranking Criteria (Priority: P2)

As a user, I can create, edit, and manage a reusable list of ranking criteria (themes, scenarios, plots) with weighted values from 1 to 5, so that I can score stories against my personal preferences. Ranking criteria are persisted independently from any story.

**Why this priority**: Ranking criteria must exist before any story can be ranked, making this a prerequisite for Story 4. It is co-prioritized with Analyze because both are enablers for the full feature value.

**Independent Test**: Can be fully tested by creating ranking criteria, editing weights, deleting criteria, and verifying persistence — all without needing a parsed story.

**Acceptance Scenarios**:

1. **Given** the user navigates to ranking criteria management, **When** a new criterion is added with a name and weight (1–5), **Then** the system persists the criterion.
2. **Given** ranking criteria exist, **When** the user views the criteria list, **Then** all criteria are displayed with their names and weights.
3. **Given** a ranking criterion exists, **When** the user edits its name or weight, **Then** the changes are persisted.
4. **Given** a ranking criterion exists, **When** the user deletes it, **Then** the criterion is removed from persistent storage.
5. **Given** the user is viewing a story detail page, **When** the user chooses to manage ranking criteria inline, **Then** the same create/edit/delete operations are available without leaving the story context.
6. **Given** a criterion weight is set outside the 1–5 range, **When** the user attempts to save, **Then** the system rejects the input with a validation message.

---

### User Story 4 - Rank a Persisted Story Against Criteria (Priority: P3)

As a user, I can rank a persisted story against my configured ranking criteria. The system uses the local LLM to score the story on each criterion, and the ranking result is persisted alongside the story metadata.

**Why this priority**: Ranking depends on both persisted stories and configured criteria, making it the highest-dependency capability. It delivers the most personalized value once the prerequisites are in place.

**Independent Test**: Can be fully tested by configuring at least one ranking criterion, opening a persisted story, invoking Rank, verifying the LLM scores each criterion, and confirming the ranking result is displayed and persisted.

**Acceptance Scenarios**:

1. **Given** a persisted story is open and ranking criteria exist, **When** the user invokes Rank, **Then** the system sends the story text and each criterion to the local LLM and returns a score for each criterion.
2. **Given** ranking completes successfully, **When** results are returned, **Then** each criterion's score and the weighted aggregate are displayed and persisted alongside the story metadata.
3. **Given** a story already has a persisted ranking result, **When** the user views the story detail, **Then** the existing ranking is displayed without requiring a new LLM call.
4. **Given** a story already has a persisted ranking, **When** the user invokes Rank again (e.g., after criteria changes), **Then** the system regenerates the ranking and replaces the previously persisted result.
5. **Given** no ranking criteria are configured, **When** the user invokes Rank, **Then** the system informs the user that criteria must be configured first and offers navigation to criteria management.
6. **Given** an LLM call for a specific criterion fails, **When** the remaining criteria succeed, **Then** the system persists and displays successful scores and shows an actionable error for the failed criterion.

---

### Edge Cases

- Story text is extremely long and may exceed the local LLM's context window.
- Story text is empty or contains only whitespace after parsing.
- Local LLM returns malformed or unparseable output for a summarize, analyze, or rank call.
- User attempts to summarize, analyze, or rank a story that has been deleted from persistence between catalog load and action invocation.
- Multiple ranking criteria have identical names.
- All ranking criteria are deleted after a story has been ranked against them.
- LLM response time exceeds a reasonable threshold, requiring user feedback on progress.
- User invokes Summarize, Analyze, or Rank while a previous request for the same story is still in progress.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST allow users to invoke Summarize on any persisted story from the story detail view.
- **FR-002**: System MUST send the full persisted story text to the local LLM as a single focused call to generate a summary.
- **FR-003**: System MUST persist the generated summary alongside the story record in SQLite.
- **FR-004**: System MUST display the persisted summary in the story detail view without requiring regeneration.
- **FR-005**: System MUST allow users to regenerate a summary, replacing the previously persisted version.
- **FR-006**: System MUST allow users to invoke Analyze on any persisted story from the story detail view.
- **FR-007**: System MUST execute four separate LLM calls for analysis — one each for characters, themes, plot structure, and writing style.
- **FR-008**: System MUST persist all analysis dimension results alongside the story record in SQLite.
- **FR-009**: System MUST display persisted analysis results organized by dimension in the story detail view.
- **FR-010**: System MUST allow users to regenerate analysis, replacing all previously persisted dimension results.
- **FR-011**: System MUST support partial-success for analysis: if some dimension calls fail, successful results are persisted and displayed while failures show actionable errors.
- **FR-012**: System MUST allow users to create, read, update, and delete ranking criteria.
- **FR-013**: Each ranking criterion MUST have a name and a weight value constrained to integers 1 through 5.
- **FR-014**: Ranking criteria MUST be persisted independently from story records and be reusable across all stories.
- **FR-015**: System MUST allow inline ranking criteria management from the story detail view.
- **FR-016**: System MUST allow users to invoke Rank on any persisted story that has at least one ranking criterion configured.
- **FR-017**: System MUST send the story text and each ranking criterion to the local LLM to produce per-criterion scores.
- **FR-018**: System MUST compute and display a weighted aggregate score from per-criterion results.
- **FR-019**: System MUST persist ranking results (per-criterion scores and weighted aggregate) alongside the story record in SQLite.
- **FR-020**: System MUST allow users to regenerate rankings, replacing previously persisted results.
- **FR-021**: System MUST support partial-success for ranking: if some criterion calls fail, successful scores are persisted while failures show actionable errors.
- **FR-022**: System MUST prevent ranking invocation when no ranking criteria are configured, with a message directing the user to criteria management.
- **FR-023**: System MUST display progress feedback to the user during LLM calls for summarize, analyze, and rank operations.
- **FR-024**: System MUST NOT overwrite existing persisted results when a regeneration attempt fails entirely.
- **FR-025**: System MUST use the existing LM Studio integration (OpenAI-compatible local LLM) for all inference calls.
- **FR-026**: Persisted feature data MUST use SQLite unless this spec explicitly states and justifies a different store.
- **FR-027**: Application logging MUST use Serilog with structured message templates and contextual properties aligned with .NET 9 logging best practices.
- **FR-028**: Major execution paths across layers/components/services MUST emit Information-level logs and provide actionable failure/error logs.
- **FR-029**: Log levels MUST be configurable via settings (including Verbose) without code changes.

### Key Entities

- **StorySummary**: A generated synopsis of a persisted story. Associated one-to-one with a ParsedStoryRecord. Contains the summary text and generation timestamp.
- **StoryAnalysis**: Structured analysis results for a persisted story. Contains four dimensions: characters, themes, plot structure, and writing style. Each dimension holds its extracted content and generation timestamp. Associated one-to-one with a ParsedStoryRecord.
- **RankingCriterion**: A user-defined criterion for scoring stories. Contains a name and a weight (integer 1–5). Persisted independently and reusable across all stories.
- **StoryRankingResult**: The outcome of ranking a story against the current set of criteria. Contains per-criterion scores and a weighted aggregate score. Associated one-to-one with a ParsedStoryRecord. Includes the generation timestamp and a snapshot of the criteria used.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of persisted stories are summarizable from the story detail view and produce a non-empty summary persisted in SQLite.
- **SC-002**: 100% of persisted stories are analyzable and produce results for all four dimensions (characters, themes, plot structure, writing style) persisted in SQLite.
- **SC-003**: Regenerating a summary or analysis for a story replaces the previous result and is reflected immediately in the detail view.
- **SC-004**: Users can create, edit, and delete ranking criteria and see changes reflected immediately in the criteria list.
- **SC-005**: 100% of ranking operations against configured criteria produce per-criterion scores and a weighted aggregate persisted in SQLite.
- **SC-006**: Partial-success scenarios for analysis and ranking persist successful results and display actionable errors for failed dimensions/criteria.
- **SC-007**: All LLM-dependent operations show progress feedback to the user during processing.
- **SC-008**: No existing persisted result (summary, analysis, or ranking) is lost when a regeneration attempt fails entirely.

## Assumptions

- Persisted stories from the StoryParser feature (001-storyparser-fetch-catalog) are available and the ParsedStoryRecord schema is the baseline for associating summary, analysis, and ranking data.
- The existing LM Studio / OpenAI-compatible client abstraction (ILmStudioClient) is sufficient for all inference calls in this feature without requiring new provider integrations.
- Four analysis dimensions (characters, themes, plot structure, writing style) are fixed for v1; additional dimensions may be introduced in a future feature.
- Ranking criteria weights are integers 1–5; fractional or extended-range weights are out of scope for v1.
- Context window management for very long stories (e.g., truncation or chunking strategies) will be addressed in the planning phase.
- The existing layered architecture, options binding, and dependency injection conventions remain the required implementation baseline.
- This feature is delivered within the StoryParser feature scope and surfaces in the existing story detail view — it does not introduce a new top-level navigation entry.
